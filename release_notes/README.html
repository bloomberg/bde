Updating Release Notes¶

These instructions assume you’re writing release notes for BDE 3.115.0.

When adding a new release notes page:

  • Begin this process on an nfs-mounted machine, because you are eventually going to want to run ‘source /bb/bde/documentation/sphinx_env/bin/activate’ which will not be available on a laptop

  • Checkout this repository

  • create a BDE-3.115.0-release-notes branch off of main

  • add the new release to the top of the list in docs/releases_notes/release_notes.rst

  • add in a blurb to the top of docs/index.rst (by copy/pasting the previous top release note and updating the version #’s and the date - note that the version number is in the blurb title AND in the link)

  • add a new docs/release_notes/bde_3.115.0.rst file, by copying bde_3.114.0.rst and fixing the version numbers, dates, highlights, and drqs list. Consult the parent DRQS for the list of the repos that went into the release and adjust this list in the release notes. Do not forget to add the new file to your git commit.

  • note that the list of repos in the release can usually be found in the main release DRQS. Remember that bde becomes bsl-internal, and bde-classic becomes bde-core when we want DPKG names rather than bbgithub repo names.

The way to generate the DRQS list is to run either {WSCH /gr 101,2217 /qu  NEXT<GO>} (if the ticket list isn’t finalized yet) or {WSCH /gr 101,2217 /qu  3.115.0<GO>}, (note that the separator between 101 and 2217 is a comma, not a period), copy and format the results from the terminal screen as described below, and then paste them into this command, not forgetting the version argument:

$ bde-docs/bin/generateDrqsList.sh 3.115.0  # <enter>, paste, then hit ^D

Remember to press <enter> before pasting the copy of the search, as it is streaming an input file, not a set of command line arguments.

To copy the search results, drag a mouse cursor to box the results from the top left of the first DRQS number, down to the bottom of the list so that it covers all of the text for each line. Then paste the results into a text file editor for formatting. If there are multiple pages of results, repeat for each page, adding to the text in the editor.

When complete, you will have text with excessive information, such as:

171873355	101	3.115.0	*F* Please address bdlf_noop nightly build failure
171801614	101	3.115.0	*F* Fix thread-naming related Windows test failure in bdlmt_eventscheduler
171797058	101	3.115.0	*F* bsls cleanup: consistency, typos etc

Use the column-editing feature of the editor to remove 101 3.115.0 *F* from all rows, producing output formatted like:

171873355	Please address bdlf_noop nightly build failure
171801614	Fix thread-naming related Windows test failure in bdlmt_eventscheduler
171797058	bsls cleanup: consistency, typos etc

Now select-all, and paste the file contents into the shell window. Finally, hit ^D to close the input buffer.

This generates everything you need to paste into the bottom of the file, replacing the old “{{{ internal” and “{{{ oss” blocks of tickets from the previous release.

Note that the oss DRQS list should be edited to remove any references to repos other than bde-tools or bde - none of the changes in other repos are visible outside of Bloomberg.

To check the rendering of your changes, follow the instructions at https://bde.bloomberg.com/bde/misc/docs-howto.html.

Writing the BLST draft¶

Run {BLST /NAME BDE <GO>} command in the terminal and use a previous BSLT entry as a template. Fill out the template with the information relevant to the current release ( collect short descriptions from the team members if necessary ) and send the draft for review to MikeV (Slava, Nathan, Joshua (?)), MikeG, OlegS.

BLST format is WYSIWYG: no backticks needed, please linkify class and component names. Some recent examples (4.15.0) have linkified the non-class-name portion of the sentance, please do not follow this example. You can link directly to class methods, as well, such as https://bde.bloomberg.com/bde-resources/doxygen/bde_api_prod/structbdlb_1_1NumericParseUtil.html#a788feadd80c9c29523232d1b7467e151. The hex identifier after the # in the URL is stable.

Once the draft is accepted, you can publish the entry:

  • Login into terminal with BDE user: {LOGA} as BDETEAM user

  • Run {BLST /NAME BDE <GO>} -> New Message

Note: if you skip the LOGA step, you will publish the BLST under your name.
